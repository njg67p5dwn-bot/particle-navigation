# Particle Navigation 기술 문서

## 개요

Particle Navigation은 마인크래프트 1.21.11 Fabric 클라이언트 모드로, 플레이어가 지정한 목표 좌표까지의 3D 최적 경로를 실시간으로 계산하고 파티클로 시각화합니다.

이 문서는 모드의 핵심 기술인 경로탐색 알고리즘, 청크 한계 우회 전략, 렌더링 방식, 그리고 성능 최적화 기법을 설명합니다.

---

## 1. 아키텍처

```
ParticleNavClient (진입점)
    │
    ├── ClientTickEvents ──→ NavigationManager.tick()
    │                            ├── AStarPathfinder.tick()   [경로 계산]
    │                            ├── PathRenderer.tick()       [파티클 렌더링]
    │                            └── showActionBar()           [HUD 표시]
    │
    └── ClientChunkEvents ──→ NavigationManager.onChunkLoad()  [경로 재계산]

NavCommand ──→ NavigationManager.setTarget() / mark() / back()
```

모든 로직은 클라이언트 측에서만 실행됩니다. 서버에 패킷을 전송하거나 서버 측 상태를 변경하지 않으므로, 멀티플레이어 서버에서도 안티치트 문제 없이 동작합니다.

## 2. 경로탐색 알고리즘 (A*)

### 2.1 A* 알고리즘 선택 이유

마인크래프트의 월드는 이산적(discrete) 3D 블록 그리드입니다. 이 환경에서 최단 경로를 찾기 위해 A* 알고리즘을 사용합니다.

| 알고리즘 | 최적성 | 효율성 | 적합성 |
|----------|--------|--------|--------|
| BFS | 최적 | 느림 (모든 노드 탐색) | 비효율적 |
| Dijkstra | 최적 | 보통 | 가중치 있으면 적합 |
| **A*** | **최적** | **빠름 (휴리스틱 유도)** | **가장 적합** |
| Greedy Best-First | 비최적 | 매우 빠름 | 최적 경로 보장 불가 |

A*는 `f(n) = g(n) + h(n)` 공식으로 노드를 평가합니다:
- `g(n)`: 시작점에서 현재 노드까지의 실제 비용
- `h(n)`: 현재 노드에서 목표까지의 추정 비용 (휴리스틱)
- `f(n)`: 총 예상 비용 — 이 값이 가장 낮은 노드를 우선 탐색

### 2.2 휴리스틱 함수

3D 유클리드 거리를 휴리스틱으로 사용합니다:

```
h(n) = sqrt((nx - gx)^2 + (ny - gy)^2 + (nz - gz)^2)
```

유클리드 거리는 admissible(실제 비용을 초과하지 않음)하므로 A*의 최적성이 보장됩니다. 맨해튼 거리 대비 더 정밀한 추정을 제공하여 불필요한 노드 탐색을 줄입니다.

### 2.3 이동 모델

마인크래프트 플레이어의 물리적 특성을 반영한 이동 규칙:

```
플레이어 히트박스: 0.6 x 1.8 x 0.6 블록
→ 간소화: 1x2x1 블록 (발 위치 + 머리 위치)
```

**이동 가능 조건 (isStandable)**

한 블록 위치에 플레이어가 설 수 있으려면:
1. 발 아래 블록(`y-1`)이 고체 (충돌 형상이 비어있지 않음)
2. 발 위치(`y`)가 통과 가능 (충돌 형상이 비어있음)
3. 머리 위치(`y+1`)가 통과 가능

```java
boolean isStandable(BlockPos feetPos) {
    return isSolidGround(feetPos.below())    // 바닥 고체
            && isPassable(feetPos)            // 발 통과 가능
            && isPassable(feetPos.above());   // 머리 통과 가능
}
```

**통과 가능 판정 (isPassable)**

블록의 실제 충돌 형상(collision shape)을 검사합니다. 이 방식은 반블록, 울타리, 유리판 등 비정형 블록도 정확히 처리합니다.

```java
boolean isPassable(BlockPos pos) {
    BlockState state = level.getBlockState(pos);
    if (state.is(Blocks.LAVA) || state.is(Blocks.FIRE)) return false;  // 위험 블록 회피
    return state.getCollisionShape(level, pos).isEmpty();              // 충돌 없으면 통과
}
```

**이동 유형별 비용**

| 이동 유형 | 비용 | 조건 |
|-----------|------|------|
| 수평 걷기 | 1.0 | 인접 4방향, 대상 위치 standable |
| 점프 (+1블록) | 1.4 | 현재 머리 위 공간 확보, 대상 standable |
| 낙하 (1~3블록) | 1.0 + 0.3*depth | 낙하 경로 전체 통과 가능, 착지점 standable |
| 사다리/덩굴 수직 이동 | 1.5 | 해당 블록이 사다리/덩굴/물 |
| 수직 하강 | 1.0 | 아래 블록이 사다리/덩굴/물 |

낙하 3블록 이하로 제한하는 이유: 마인크래프트에서 3블록 이하 낙하는 데미지가 없고, 4블록부터 낙하 데미지가 발생합니다.

### 2.4 이웃 노드 확장

각 노드에서 최대 14개의 이웃을 검사합니다:

```
수평 4방향 × (걷기 + 점프 + 낙하 1~3) = 최대 4 × 3 = 12
수직 이동 (올라가기 + 내려가기)           = 2
                                    합계 = 최대 14
```

실제로는 대부분의 방향이 조건을 만족하지 못하므로, 평균적으로 노드당 3~5개의 유효 이웃이 생성됩니다.

## 3. 시간 분할 실행 (Time-Slicing)

### 3.1 문제

A* 알고리즘은 장거리 경로 계산 시 수만~수십만 개의 노드를 처리해야 합니다. 이를 한 프레임에서 처리하면 심각한 프레임 드롭이 발생합니다.

마인크래프트의 게임 루프는 1틱 = 50ms(20TPS)이며, 이 중 경로 탐색에 할당할 수 있는 시간은 극히 제한적입니다.

### 3.2 해결: 틱 예산 기반 분할

경로 탐색을 한 번에 완료하지 않고, 매 틱마다 정해진 시간 예산(3ms) 내에서만 노드를 처리합니다.

```
틱 1:  [시작] → 200개 노드 처리 → 시간 초과 → 일시 중지
틱 2:         → 200개 노드 처리 → 시간 초과 → 일시 중지
틱 3:         → 150개 노드 처리 → 목표 도달 → 경로 완성
```

```java
public boolean tick(long maxMs) {
    long deadline = System.nanoTime() + maxMs * 1_000_000;

    while (!openSet.isEmpty() && System.nanoTime() < deadline) {
        // 노드 처리...
    }

    return finished;
}
```

### 3.3 시간 예산 선택 근거

| 예산 | 틱당 처리 노드 | 50블록 경로 소요 | 체감 렉 |
|------|---------------|-----------------|---------|
| 1ms | ~60개 | ~15틱 (0.75초) | 없음 |
| **3ms** | **~200개** | **~5틱 (0.25초)** | **거의 없음** |
| 10ms | ~600개 | ~2틱 (0.1초) | 약간 |
| 50ms | ~3000개 | 1틱 | 프레임 드롭 |

3ms는 50ms 틱의 6%만 차지하므로, 나머지 94%는 렌더링과 게임 로직에 사용됩니다.

### 3.4 상태 유지

A*의 상태(openSet, allNodes, bestNode)를 인스턴스 변수로 유지하여, 매 틱에서 이전 상태를 이어서 처리합니다.

## 4. 청크 로딩 한계와 점진적 탐색

### 4.1 문제

마인크래프트 클라이언트는 서버로부터 렌더 거리 내의 청크만 수신합니다. 렌더 거리 16청크 기준, 로드 범위는 약 256블록입니다. 이 범위 밖의 블록 정보는 존재하지 않으므로, 장거리 경로를 한 번에 완전히 계산하는 것은 불가능합니다.

```
    로드된 영역 (256블록)
   ┌─────────────────────┐
   │  [플레이어]          │         [목표: 1000블록 떨어짐]
   │     ↓                │              ↑
   │  A* 정밀 탐색 가능    │         블록 정보 없음
   └─────────────────────┘
```

### 4.2 해결: 확정 경로 + 추정 경로

경로를 두 구간으로 나눕니다:

1. **확정 경로 (Confirmed)**: 로드된 청크 내에서 A*로 계산한 정확한 경로
2. **추정 경로 (Estimated)**: A* 탐색 끝점에서 목표까지의 직선 보간 경로

```java
// A*가 목표에 도달하지 못한 경우 (청크 밖)
if (!reachedGoal) {
    // 탐색된 노드 중 목표에 가장 가까운 노드에서
    // 목표까지 직선 보간 경로를 추가
    addEstimatedPath(path, bestNode.pos, goal);
}
```

추정 경로는 시작점과 끝점 사이를 선형 보간(linear interpolation)하여 생성합니다:

```
추정 경로 점 = from + (to - from) × (i / steps)
```

### 4.3 점진적 재계산

플레이어가 이동하면서 새로운 청크가 로드될 때, 추정 경로를 확정 경로로 교체합니다.

```
시점 1: [플레이어] ===확정=== [청크 경계] ---추정--- [목표]
            ↓ (플레이어 이동, 새 청크 로드)
시점 2:          [플레이어] ===확정=== [새 경계] ---추정--- [목표]
            ↓ (반복)
시점 N:                              [플레이어] ===확정=== [목표]
```

**재계산 트리거 조건:**

| 트리거 | 조건 | 쿨다운 |
|--------|------|--------|
| 청크 로드 | 추정 경로가 해당 청크를 통과 | 40틱 (2초) |
| 경로 이탈 | 플레이어가 경로에서 5블록 이상 벗어남 | 40틱 (2초) |

쿨다운을 두는 이유: 플레이어가 빠르게 이동하면 여러 청크가 연속 로드되는데, 매번 재계산하면 이전 계산이 완료되기 전에 새 계산이 시작되어 CPU가 낭비됩니다.

```java
public void onChunkLoad(int chunkX, int chunkZ) {
    if (recalcCooldown > 0) return;  // 쿨다운 중이면 무시

    for (PathNode node : currentPath) {
        if (!node.confirmed()) {  // 추정 경로 노드만 검사
            if (해당 청크에 속하는가) {
                startPathfinding();       // 현재 위치에서 재계산
                recalcCooldown = 40;      // 2초 쿨다운
                break;
            }
        }
    }
}
```

## 5. 파티클 렌더링

### 5.1 파티클 종류

`DustParticleOptions`(레드스톤 먼지 파티클)을 사용하며, RGB 색상과 크기를 지정합니다:

| 용도 | 색상 코드 | 크기 | 시각적 의미 |
|------|----------|------|------------|
| 확정 경로 | `0x00FF33` (초록) | 1.2 | A*로 검증된 안전한 경로 |
| 추정 경로 | `0xFFFF00` (노란) | 1.0 | 직선 추정, 실제 통행 불가능할 수 있음 |
| 목표 지점 | `0xFF3300` (빨강) | 1.5 | 목표 위치 비컨 기둥 |

### 5.2 렌더링 최적화

**렌더 거리 제한**: 플레이어로부터 60블록 이내의 경로 노드만 파티클을 생성합니다.

**틱 간격 조절**: 매 틱이 아닌 3틱마다 파티클을 생성합니다. 마인크래프트의 먼지 파티클은 수 틱 동안 지속되므로, 시각적 연속성이 유지됩니다.

**보간 파티클**: 경로 노드 간 거리가 1.5블록 이상인 경우, 중간에 보간 파티클을 추가하여 끊김 없는 연속적인 경로 표시를 구현합니다.

```
노드 간 거리 > 1.5 → 중간 보간점 생성:

  [노드A] ·  ·  ·  [노드B]    (보간 파티클로 연결)
```

### 5.3 가장 가까운 노드 탐색

매 렌더링 시, 플레이어에게 가장 가까운 경로 노드를 찾아 해당 지점 주변부터 렌더링합니다. 이미 지나온 경로를 과도하게 렌더링하는 것을 방지합니다.

## 6. 방향 표시 (HUD)

### 6.1 상대 각도 계산

플레이어의 시점 방향과 목표 방향 사이의 상대 각도를 계산하여 방향 화살표를 표시합니다.

```
1. 목표 방향 벡터: toTarget = normalize(goal - playerPos)
2. 목표의 절대 각도: targetAngle = atan2(-toTarget.x, toTarget.z)
3. 플레이어의 시점 각도: playerYaw (게임에서 제공)
4. 상대 각도: relativeAngle = (targetAngle - playerYaw) mod 360
```

### 6.2 8방향 화살표 매핑

상대 각도를 45도 단위로 8방향 화살표에 매핑합니다:

```
         337.5° ~ 22.5°
              ↑
    292.5°  ↖   ↗  67.5°

    247.5°  ←   →  112.5°

    202.5°  ↙   ↘  157.5°
              ↓
```

화살표는 액션바(핫바 위)에 거리 정보와 함께 표시됩니다:

```
↗ 142.3m              (이동 중)
↗ 142.3m [탐색중...]  (경로 계산 중)
  목표 도착!           (3블록 이내 도달)
```

## 7. 스레드 안전성

모든 경로 탐색과 렌더링은 **메인 스레드(렌더 스레드)**에서 실행됩니다. 이는 의도적인 설계입니다:

- 마인크래프트의 `ClientLevel` 블록 데이터는 메인 스레드에서만 안전하게 접근 가능
- 별도 스레드 사용 시 블록 데이터의 스냅샷이 필요하여 메모리 비용 증가
- 시간 분할로 충분한 성능을 확보하므로 멀티스레딩의 필요성이 낮음

경로 결과는 `CopyOnWriteArrayList`에 저장하여, 경로 재계산 중에도 렌더러가 이전 경로를 안전하게 읽을 수 있습니다.

## 8. 탐색 한계 및 안전장치

### 8.1 최대 반복 횟수

A* 탐색은 최대 100,000개 노드로 제한됩니다. 이는 경로를 찾을 수 없는 경우(예: 목표가 밀폐된 공간 내부) 무한 탐색을 방지합니다.

한계에 도달하면, 탐색된 노드 중 목표에 가장 가까운 노드까지의 경로를 반환하고, 나머지는 추정 경로로 대체합니다.

### 8.2 위험 블록 회피

경로 탐색에서 다음 블록은 통과 불가능으로 처리합니다:
- **용암**: 접촉 시 화염 데미지
- **불**: 접촉 시 화염 데미지
- **달콤한 열매 덤불**: 접촉 시 이동 속도 감소 + 데미지

### 8.3 부분 경로 (Partial Path)

목표에 도달할 수 없을 때, 가장 가까이 접근한 노드까지의 확정 경로 + 나머지 추정 경로를 반환합니다. 이를 통해 "경로 없음" 대신 "최선의 방향"을 제시합니다.

## 9. 데이터 구조

### 9.1 PathNode

```java
record PathNode(BlockPos pos, boolean confirmed) {}
```

경로의 각 지점을 나타내는 불변 레코드. `confirmed`가 `true`이면 A*로 검증된 경로, `false`이면 직선 추정입니다.

### 9.2 A* 내부 Node

```java
class Node {
    BlockPos pos;     // 블록 좌표
    Node parent;      // 경로 추적용 부모 노드
    double g;         // 시작점부터의 실제 비용
    double h;         // 목표까지의 휴리스틱
    double f;         // g + h
    boolean closed;   // 탐색 완료 여부
}
```

### 9.3 자료구조 선택

| 용도 | 자료구조 | 이유 |
|------|---------|------|
| Open Set | `PriorityQueue<Node>` | f값 기준 최소 노드를 O(log n)으로 추출 |
| 노드 조회 | `HashMap<Long, Node>` | BlockPos.asLong()을 키로, O(1) 조회 |
| 경로 결과 | `CopyOnWriteArrayList` | 읽기/쓰기 동시 접근 안전 |

## 10. 한계와 향후 개선 가능성

| 한계 | 원인 | 개선 방향 |

| 위치 저장 휘발성 | 메모리에만 저장 | 파일 기반 웨이포인트 시스템 |
| 네더 포탈 경유 불가 | 차원 간 경로 탐색 미지원 | 포탈 좌표 변환 + 다차원 그래프 |
